Simple explanation with a real-world analogy:

Block Storage = Hard Drive on your computer
Think of it like a physical hard disk attached to your server. Your app, database files, code, operating system — everything lives here. You can read, write, edit files freely. Fast and works like normal storage.
* Your BharatScan app → stored here
* Your SQLite database (stock data) → stored here
* 200 GB is your main working space

Object Storage = Google Drive / Dropbox in the cloud

* You could dump old CSV backups here
* Or nothing at all — you may never even need it
* 10 GB is just a bonus on the side

For your use case — only Block Storage matters.
You'll use the 200 GB block storage for everything:
* The app itself (~500 MB)
* Stock database growing over time (~5–20 GB for years of NSE data)
* Options data, logs, backups
You have plenty of room. Object storage is just extra that you can ignore for now.

Deciding on explaining Static IP
Good question. Let me explain it simply:

What is an IP Address?
Every server on the internet has an IP address — think of it like a home address for your server. Example: 143.47.82.21

Dynamic IP vs Static IP
Dynamic IP:
* Your server's address changes occasionally (every restart, or randomly)
* Like living in a house where your door number keeps changing
* Problem: if someone bookmarked your old address, they can't find you anymore
Static IP:
* Your server's address is fixed, never changes
* Like a permanent address that always works
* Anyone can always reach your server at the same address

Do You Need a Static IP?
For your setup — yes, but it's already included free on Oracle ✅
Oracle Cloud gives you a free Reserved Public IP (their term for Static IP) — you assign it to your server once and it never changes, even if you restart the server.

But Here's the Better Solution — A Domain Name
Instead of sharing an ugly IP like 143.47.82.21, you'd get a proper domain name like:
* bharatscan.in (~₹800/year)
* bharatscan.com (~₹1000/year)
* or even a free subdomain
You point the domain to your Oracle Static IP once. From then on, you open bharatscan.in in any browser — done. Even if you ever move to a different server, just update where the domain points. Users never notice.

Summary for Your Setup

For personal use: Static IP alone is fine. Just bookmark your IP address. For public users: A proper domain name is essential — people won't remember a raw IP number.

Planning feedback review and explanation
Claude gave excellent, specific feedback. Here's what it means in plain language:

Things in the Wrong Order
CORS locking too early: In Phase 1 we said "lock CORS to your domain" — but at that point you don't have a domain yet. The Cloudflare tunnel URL only comes in Phase 3. So locking CORS in Phase 1 would be pointing to nothing. Move this to Phase 3 after the tunnel is set up.
Database migration system too late: We put this in Phase 4, but Claude correctly points out — every time you add a new table or column in Phase 2, you need a migration system already in place. Otherwise when you make schema changes on Oracle, you risk wiping the market data you've been accumulating. Move this to the very beginning of Phase 2.

Things Missing from Phase 3 (Oracle Setup)
pnpm not mentioned: Oracle's fresh Ubuntu server has npm but not pnpm. Your entire project runs on pnpm. Without explicitly installing pnpm first, nothing will install. Easy to forget, must be in the setup steps.
Oracle firewall not mentioned: Oracle blocks ALL ports by default — only SSH (port 22) is open. Your app will run fine on the server but no browser in the world can reach it until you manually open your app's port in Oracle's dashboard. Critical step that was completely missing from our plan.
.env file not mentioned: The .env file (which holds your API keys, database path, server port) is intentionally excluded from GitHub (correctly). So when Oracle pulls your code from GitHub, the .env doesn't come with it. Someone has to manually create it on Oracle. Easy to forget and the app will crash without it.
Cloudflare Tunnel needs to stay running permanently: The tunnel process needs to keep running even after server restarts. Either PM2 or Ubuntu's systemd service needs to manage it. Not mentioned anywhere — without this, every time Oracle restarts, your URL goes dead.

Things Missing from Phase 2
NSE holiday calendar for the scheduler: The auto-sync scheduler should automatically know not to run on weekends and market holidays. Currently not mentioned. Without this, it will try to fetch data on days the market is closed and either fail or fetch empty/wrong data.
Angel TOTP secret extraction — the actual steps: We said "automate TOTP token refresh" but didn't explain how to actually get the raw TOTP secret from Angel's setup. It's not obvious — the QR code Angel shows during 2FA setup contains the raw secret, and you need to extract it before completing 2FA setup. If you complete setup without saving it, you'd need to reset 2FA. This is a specific non-obvious step that needs to be documented.
Sensex options on BSE — needs verification first: Sensex options trade on BSE, not NSE. Angel API may or may not provide them in the same format as Nifty options. Claude's point: verify this works with Angel BEFORE building the Sensex options sync. Otherwise you build the whole thing and discover it silently returns no data.

Phase 2 is too large: Claude correctly points out that Phase 2 has 16+ items — Angel auth, rate limiting, 6 data fetchers, scheduler, caching, ATM calculator, alerting, provider layer — this is realistically weeks of work. Either split it into sub-phases or at least acknowledge it'll take time. Good to set realistic expectations.
Caching layer is premature: SQLite running on a local server is already very fast for a single user. Building a caching layer before you've actually experienced any slowness adds unnecessary complexity. Claude's suggestion: skip caching for now, add it only when you actually see slowness in production. Sensible advice.

Things That Will Actually Break Deployment
1. Git authentication from Oracle to GitHub is missing When GitHub Actions SSHes into Oracle and runs git pull, Oracle needs permission to actually pull from your GitHub repo. If the repo is private, git pull silently fails or sits waiting for a password — your deployment hangs with no clear error.
The fix is a GitHub Deploy Key:
* Generate a second SSH key pair (different from the GitHub Actions key)
* Add the public key to GitHub repo → Settings → Deploy Keys
* Add the private key to Oracle's ~/.ssh
This step goes in Phase 3 between "Clone app from GitHub" and "Create .env file."
2. pnpm install should be pnpm install --frozen-lockfile Without this flag, pnpm can silently update the lockfile during a production deployment — potentially pulling in untested or newer package versions that break things. Should be this flag in both deploy.yml AND post-merge.sh.

Thing That Will Corrupt Backups
3. Copying SQLite WAL files with cp can still produce a corrupt backup Even if all 6 files are copied together, doing it while the database is actively running can still corrupt the backup. The correct approach is either:
* Use SQLite's built-in .backup command, OR
* Run PRAGMA wal_checkpoint(FULL) first to flush all WAL data into the main .db file, then copy
The backup script in Phase 2D needs to use one of these — not a plain file copy.

Thing That Will Cause Confusion During Oracle Setup
4. Cloudflare Tunnel login on a headless server is non-obvious Running cloudflared tunnel login on Oracle opens a browser URL — but Oracle has no browser. It prints a URL on screen and waits. You must copy that URL, open it in your own computer's browser, authorise it there, and then the Oracle terminal continues. If you don't know this, the setup just stalls silently. Needs a clear note in Phase 3.

Important Code Change Hidden in Phase 2A
5. Splitting the database is a much wider code change than it sounds Currently server/src/db.ts exports a single database connection that every route file uses. After the split into market.db and app.db, every single route file — scans.ts, portfolio.ts, alerts.ts, settings, etc. — needs to be updated to import from the correct database connection.
This is not just "create two files." It's a refactor of every route file in the server. Needs to be listed explicitly in Phase 2A so it doesn't get missed.

Minor
6. Node.js must be v22.5 or later — not just "v22" Node v22.0 through v22.4 do NOT include node:sqlite. Someone installing Node 22.0 will still get a startup failure. Must specify v22.5+ minimum.
7. PM2 restart in deploy.yml should use the app's specific process name If pm2 restart all is used, it works — but later when Cloudflare Tunnel is also running as a PM2 process, every deploy briefly kills the tunnel too. The PM2 process name in ecosystem.config.js and the pm2 restart <name> in deploy.yml should match exactly from the start.

1. Server timezone must be set to IST — or all sync times will be wrong Oracle's Ubuntu image defaults to UTC like almost every cloud provider. Explaining Claude's suggestions
Claude says this round did an excellent job — almost everything is solid. Three specific things still missing:

The Most Important One — Start Oracle NOW, In Parallel
The phase sequencing is wrong for one critical reason: Oracle's 7-day idle reclaim policy.
Currently the document says: finish Phase 1 → finish Phase 2A through 2D → then do Phase 3 (Oracle setup). That means you'd grab the Oracle instance weeks from now, after all the code is done.
The problem: Oracle can reclaim idle free-tier instances after 7 days of inactivity. If you create the instance and then don't touch it while doing weeks of Phase 1 and 2 coding, Oracle may stop or terminate it before you're ready to use it.
There's also a second problem: getting an ARM instance in the first place can take hours to days of retrying. Oracle free-tier ARM capacity is limited by region. You may click "Create" and get "out of capacity" errors repeatedly before it succeeds. This is completely normal — but it means instance creation itself could block you for days if you wait until the code is ready.
The fix: Pull a small set of Oracle steps out and run them as a parallel track starting today — completely independent of the code work:
1. Create Oracle account now
2. Start retrying ARM instance creation now (may take multiple attempts over hours/days — that's expected, document this)
3. Once instance exists: install Node v22.5+, pnpm, PM2
4. Run a trivial placeholder process under PM2 (even just pm2 start echo or a simple hello-world) — this counts as "active" and prevents Oracle from reclaiming the instance while Phase 1 and 2 coding happens on Replit
5. Keep the instance "warm" with occasional activity until the real app is ready to deploy
This doesn't require renumbering all the phases — just create a "Phase 0: Oracle Instance (Start Immediately, Parallel)" section.

The Central Scheduler Needs Its Own Separate PM2 Process
Currently the plan implies the Central Scheduler (which runs the 4 PM sync, intraday sync, options sync, nightly cleanup) lives inside the same Node process as the Express server. That's the natural way to build it — but it creates a serious problem:
Every time GitHub Actions deploys a code change, it runs pm2 restart bharatscan. If the scheduler is inside the same process, that restart kills the scheduler mid-run. A 4 PM EOD sync running across 2000+ stocks could be interrupted by a routine deploy at 4:20 PM and the data for that day would be incomplete.
The fix: The scheduler must be a completely separate PM2 process with its own name — e.g. bharatscan-scheduler. Two entries in ecosystem.config.js:
* bharatscan — the Express web server
* bharatscan-scheduler — the Central Scheduler
Deploys only restart bharatscan. The scheduler keeps running uninterrupted. If the scheduler itself needs updating, it gets its own deliberate restart, not an accidental one from every code deploy.

Auto-Rollback Has Nothing Concrete to Roll Back To
The deployment pipeline says "if health check fails → auto-rollback." But the deploy script as written never captures what the previous working commit was before pulling new code. So "rollback" has no actual target.
The fix: The deploy script must store the current commit hash into a variable right before git pull:
PREV_COMMIT=$(git rev-parse HEAD)
git pull
pnpm install --frozen-lockfile
pnpm build
pm2 restart bharatscan
# health check...
# if fails:
git reset --hard $PREV_COMMIT
pnpm build
pm2 restart bharatscan

Without saving PREV_COMMIT before the pull, the rollback command has an undefined target and won't work.
