Bug: Symbol Master only ever syncs on a "every Monday 7 AM IST" cron 
(config.syncSchedule.symbolMaster in environment.ts). A fresh deployment 
that goes live on any day other than Monday morning sits with a 
completely empty `symbols` table until the following Monday. Since EOD 
sync, intraday sync, F&O auto-subscribe, and ATM option tracking all 
query `symbols` first to know what to work on, every one of those jobs 
silently does nothing in the meantime — and since none of them have ever 
succeeded even once, catchUpScheduler's startup catch-up also skips them 
(it only walks forward from a job's last successful run, so a job with 
zero history gets no help from catch-up either). Net effect: a brand-new 
deployment can sit fully idle for up to 6 days with no automated data at 
all, and the user has to know to manually click "Refresh Symbol Master" 
in Settings to unblock everything downstream.

Note: syncSymbolMaster() in server/src/services/symbolMasterService.ts 
downloads Fyers' public CM/F&O symbol master files and NSE's index pages 
directly — it does NOT require an authenticated broker session. So this 
bootstrap can and should run on every server start, independent of 
whether a broker is currently connected.

Fix, in server/src/services/scheduler.ts:

1. Add a new function `bootstrapSymbolMasterIfEmpty(): Promise<void>` 
   (co-locate it in scheduler.ts, or in symbolMasterService.ts and import 
   it — your call, just keep it next to syncSymbolMaster's existing 
   logging style). It should:
   a. Query `SELECT COUNT(*) as c FROM symbols` on marketDb.
   b. If count > 0, log nothing (or a quiet debug line) and return 
      immediately — this must be a no-op on every normal restart once 
      the table is populated, so it never fights the Monday cron or 
      duplicates work.
   c. If count === 0, log 
      "[scheduler] symbols table is empty — running one-time bootstrap 
      symbol master sync before registering cron jobs …", call 
      syncSymbolMaster(marketDb), and log the upserted count on success 
      (reuse the same log format as the existing Monday cron handler in 
      scheduler.ts).
   d. Wrap the syncSymbolMaster() call in try/catch — if it fails (e.g. 
      Fyers' static file endpoints are briefly unreachable), log the 
      error clearly and return normally rather than throwing. Do NOT let 
      a bootstrap failure crash server startup. The table stays empty, 
      the next restart (or Monday's cron, or the manual Settings button) 
      will retry.

2. Call `await bootstrapSymbolMasterIfEmpty()` inside `registerAllCronJobs()` 
   (the function created in the previous prompt), as the very first thing 
   it does — before the `void runStartupCatchUp()` call and before any 
   cron.schedule(...) registration. Await it (don't fire-and-forget) so 
   that by the time catch-up and the rest of the cron jobs are registered, 
   symbols are already there if the download succeeded. This must run 
   through the same code path on Replit, local/Electron, AND Oracle's 
   standalone.ts — since registerAllCronJobs() is already the shared 
   entry point for all three after the last fix, no per-platform changes 
   should be needed here.

3. Do not touch the existing Monday 7 AM cron registration, the manual 
   POST /api/symbols/refresh route, or syncSymbolMaster()'s internals — 
   this is purely an additional one-time guard at startup, not a 
   replacement for the weekly refresh.

After this change, confirm with a fresh/empty database (or by manually 
deleting all rows from `symbols` in market.db and restarting the dev 
server) that: symbols gets populated automatically on startup with no 
manual click required, the log line appears exactly once, and a second 
restart with symbols already populated does NOT re-run the sync.

Do not change anything about historical price backfill (ohlcv_daily) in 
this prompt — that's a separate, deliberately manual step via "Load from 
connected broker" in Settings, and should stay that way unless explicitly 
asked to automate it too.
