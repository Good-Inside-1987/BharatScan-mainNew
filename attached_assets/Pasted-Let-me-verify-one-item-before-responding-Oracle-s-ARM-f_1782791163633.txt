Now, the document itself — everything from round 6 is correctly incorporated. A few things still missing:
Sync jobs may fire at the wrong time of day. Oracle's Ubuntu image, like nearly every cloud provider's base image, defaults to UTC. The entire daily schedule — 4:00 PM, 4:30 PM, 5:00 PM, 5:30 PM — is written assuming IST. If the scheduler's cron expressions aren't explicitly told to use Asia/Kolkata, or the server's system timezone isn't set to it, "4 PM" will actually fire at 4 PM UTC, which is 9:30 PM IST — hours after market close and after you'd want EOD data ready. Phase 3 needs an explicit step: either sudo timedatectl set-timezone Asia/Kolkata on the server, or hardcode IST in the scheduler library regardless of system time. Nowhere across seven rounds has this been addressed.
A broken build could silently replace a working app. The pipeline diagram has said "health check confirms app is live" since round one, but it's never once been assigned as an actual task in any phase — it's a label in a diagram, not a build item. Relatedly, the deploy script as described (git pull → pnpm install → pnpm build → pm2 restart) has no failure handling. If pnpm build fails on a bad push, a script without set -e (or equivalent) will continue to the restart step anyway, potentially leaving Oracle running a half-built or broken app. The deploy.yml needs to stop immediately on any failed step, and after restart, actually curl the existing /api/health endpoint with a couple of retries — if it doesn't return 200, fail the GitHub Action and fire the Telegram alert rather than reporting success.
Telegram, not email, for the failure alert. The document still says "Telegram or email" without deciding. Worth deciding now: cloud providers commonly block outbound SMTP port 25 by default on free-tier accounts specifically to prevent spam abuse, and even where it's open, mail sent cold from a cloud VM's IP often lands in spam or gets rejected by Gmail/Outlook due to poor sender reputation on those IP ranges. Telegram's Bot API is just an HTTPS POST, so it sidesteps this entirely. Setup is a five-minute step — message @BotFather on Telegram to get a bot token, then get your own chat ID — but it should be the explicit choice in Phase 2D rather than left as an "or."
