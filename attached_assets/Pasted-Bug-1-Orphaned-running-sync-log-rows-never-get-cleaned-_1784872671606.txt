Bug 1 — Orphaned "running" sync_log rows never get cleaned up
File: server/src/services/syncJobs.ts (or wherever sync_log rows are managed at startup)

When the app is closed while a job is mid-run, its sync_log row stays status='running'
forever (no finished_at). Confirmed example: id=14, job_name='eod_sync',
target_date='2026-07-22', started_at='2026-07-23T12:47:55', still 'running'.

Fix: on server startup (before catchUpScheduler runs), find any sync_log rows still
status='running' and mark them status='failed' with error_message='Orphaned — process
restarted mid-job' and finished_at=now. Since this is a single-instance Electron app
(only one process ever writes to this DB at a time), any 'running' row found at startup
is guaranteed stale — no need for an age threshold, just clean all of them found at boot.

Bug 2 — EOD sync stats mismatch (completed=0 despite many visible successes)
File: server/src/services/syncJobs.ts, runSymbolLoop()

In production log from July 23 (sync_log id=15), the job printed hundreds of lines like:
  [marketDataService] ✓ inline fetch NSE:XXX-EQ 1D 2026-07-22→2026-07-22  (1 bars)
yet the final summary was:
  EOD sync done — status=failed completed=0 skippedBudget=0 failed=2964
completed=0 with failed=total symbols is inconsistent with the visible per-symbol
successes — every success should have hit `stats.completed++` in runSymbolLoop's
success branch (bars.length > 0).

Please do NOT guess-fix this. First add temporary diagnostic logging inside
runSymbolLoop's loop body:
  - Log a running tally every 200 symbols processed: 
    console.log(`[syncJobs][debug] progress: completed=${stats.completed} 
    skippedBudget=${stats.skippedBudget} failed=${stats.failed} at symbol #${i}`)
  - Also log immediately any time bars.length > 0 but stats.completed does NOT get 
    incremented right after (add an assertion-style check right after the increment 
    to confirm it actually ran), so if there's a silent early-return, exception 
    swallow, or shadowed variable, it surfaces immediately in the log rather than 
    only showing up in the final mismatched summary.

Report back the exact point where completed stops incrementing despite visible bar
data, so the real fix can be scoped precisely rather than guessed at.
