The Scanner (artifacts/bharatscan/src/pages/Index.tsx and ScannerDashboard.tsx) currently only
gets data from a manual CSV upload or a manual "Load from connected broker" click each session —
it has zero connection to the ohlcv_daily table that the nightly EOD sync job (built earlier)
already fills in automatically every trading day at 4pm. Bridge these together so opening the
Scanner just works with already-synced data, no manual action needed.

1. Add a new read-only route, e.g. GET /api/scanner/universe-data?from=YYYY-MM-DD&to=YYYY-MM-DD
   (default range: last 2 years if not specified), that reads DIRECTLY from the local ohlcv_daily
   table (via a plain SQL query, no Fyers API calls, no daily request budget consumption at all —
   this must never touch getHistoricalBars/marketDataService's budget system since it's just
   reading what nightly sync already cached locally).

2. Return every symbol that has at least one ohlcv_daily row in range, shaped to exactly match the
   existing SymbolHistory[]/Bar[] types already used by the Scanner (see
   artifacts/bharatscan/src/lib/csv.ts):
     interface Bar { date, open, high, low, close, prevClose, volume, trades, value, strike? }
     interface SymbolHistory { symbol, series, bars: Bar[] (sorted ascending by date), expiry? }
   Notes on fields that don't exist in ohlcv_daily:
   - prevClose: compute it server-side from each symbol's own previous row by date — don't leave
     it blank/zero except for a symbol's very first available bar.
   - series: use "EQ" as a reasonable default (matches standard NSE bhavcopy convention) unless the
     symbols table already has better series info to use instead.
   - trades and value: broker-sourced data doesn't include these (bhavcopy CSVs did) — default
     both to 0 rather than guessing, and don't let this cause errors anywhere in the existing
     scan/indicator engine that might reference them; if any existing indicator specifically
     depends on trades or value being non-zero, leave it working exactly as it does today for CSV
     data, just now also tolerating 0 gracefully for broker-sourced rows.

3. In artifacts/bharatscan/src/pages/Index.tsx (or wherever the Scanner's data context
   initializes), on first load: if "histories" is still empty (nothing loaded via CSV or manual
   broker-load yet this session) AND the broker is connected, automatically call this new endpoint
   in the background and populate histories via the existing setHistories() the same way
   handleLoadFromBroker already does. Show a small, non-blocking loading indicator/toast ("Loading
   cached market data…") rather than a full-page blocker, since this should feel instant compared
   to the manual broker-load flow.

4. Add a manual "Refresh from Database" button near the existing Data Source panels (Settings →
   API/Data Source, next to the existing CSV upload and broker-load options) that re-triggers the
   same fetch on demand — useful if someone leaves the app open across the 4pm nightly sync and
   wants today's freshly-synced data without reloading the whole page.

5. If the new endpoint returns zero symbols (nothing synced yet — e.g. first run, before any
   nightly job has completed), fall back gracefully to the existing "No data loaded yet — upload a
   CSV or connect a broker" empty state already shown today — don't show an error, just don't
   auto-populate anything.
