Bug: the startup catch-up orchestrator (runStartupCatchUp in
server/src/services/catchUpScheduler.ts) skips a job entirely if it has
NEVER completed successfully:

    const lastSuccess = getLastSuccessDate(job.jobName);
    if (!lastSuccess) continue; // never run before — nothing to "catch up" from

This is the exact same failure class as the symbol-master empty-table bug
fixed earlier (bootstrapSymbolMasterIfEmpty in scheduler.ts): walking
forward from a last-success date that doesn't exist means there's nothing
to walk forward *from*, so eod_sync / intraday_sync / options_sync /
fo_ban_list / supplementary get skipped by catch-up forever on a fresh
deployment (or any deployment where these jobs have literally never
succeeded once). The normal cron jobs still fire at their scheduled times
going forward, but nothing backfills the gap automatically — the only way
data appears is if the server happens to already be running through that
job's exact scheduled time on a weekday, or via the manual "Load from
connected broker" / CSV upload panel. That's why the Scanner (and other
pages) end up asking for CSV/manual broker load even though Fyers is
connected and the automation is otherwise wired up correctly: ohlcv_daily
(and the other synced tables) are simply empty because catch-up never
bootstrapped them.

Fix, in server/src/services/catchUpScheduler.ts, inside
runStartupCatchUp()'s backlog-building loop:

1. Add a small helper above listMissedWeekdays:

    /** Walks backward from `dateStr` (inclusive) to the nearest actual trading day. */
    function mostRecentTradingDay(dateStr: string): string {
      let d = dateStr;
      while (!isTradingDay(d)) d = addDays(d, -1);
      return d;
    }

2. Replace the `if (!lastSuccess) continue;` line with logic that
   bootstraps the job using the most recent trading day instead of
   skipping it:

    if (!lastSuccess) {
      const bootstrapDate = mostRecentTradingDay(yesterday);
      if (!hasSuccessfulRunForDate(job.jobName, bootstrapDate)) {
        backlog.push({ job, date: bootstrapDate });
      }
      continue;
    }

   Leave the existing `missed`/`listMissedWeekdays` branch below unchanged
   for jobs that DO have a lastSuccess date.

3. Do NOT attempt a deep multi-year backfill here — that stays the
   deliberate manual "Load from connected broker" flow (same as the
   Symbol Master bootstrap prompt's note). This fix only guarantees each
   job gets at least one real day of data automatically so the pipeline
   (and the Scanner's auto-populate-from-database bridge) actually has
   something to work with; the normal 4pm/4:30pm/5pm cron jobs (and this
   same catch-up logic on future restarts) take over accumulating more
   history from there.

Do not change trigger B (runPeriodicCatchUpCheck) — it doesn't have this
bug, since it checks "has today's scheduled time passed with no
successful run logged for today" directly, without depending on
getLastSuccessDate.

recent trading day (check server logs for "Catching up %s for %s"), that
