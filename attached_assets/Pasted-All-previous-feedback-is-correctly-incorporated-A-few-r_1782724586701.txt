All previous feedback is correctly incorporated. A few remaining gaps:
Will Actually Block Something
Oracle Object Storage needs its own setup that isn't mentioned anywhere. Creating the bucket, generating Object Storage access credentials, and configuring the backup script to authenticate with Object Storage are separate steps from just "having an Oracle account." Without these, the nightly backup command has nowhere to write to. This belongs in Phase 2D when the backup script is built, and again in Phase 3 as a setup step.
The single DB → two DB split has a transition problem. Phase 1 adds indexes and moves the API key to the existing single bharatscan.db. Phase 2A then splits into market.db and app.db. But the data already written in Phase 1 (saved scans, portfolios, alerts, settings) is sitting in bharatscan.db. The migration script that moves existing app data into app.db is not mentioned. Without it, all Phase 1 data is silently abandoned when Phase 2A runs.
Will Cause a Subtle Planning Problem
Angel SmartAPI's 1-minute intraday history limit is closer to 30 days, not 60-90 days. The Day 1 backfill strategy says "intraday: last 60-90 days from Angel API" but this may not be available. Worth verifying with actual Angel API documentation before building the backfill script, otherwise the first-run sequence will fail silently or return less data than expected.
The 10-year EOD backfill for 2000+ stocks with rate limiting will take several hours on Day 1. If the Oracle server reboots or connection drops at hour 3, the script has no checkpoint — it restarts from the beginning. The first-run backfill script needs a simple progress marker (last symbol + date successfully processed) so it can resume from where it stopped rather than starting over.
Small but Worth Noting
pm2 startup is a two-step process. Running it prints a command (usually starting with sudo env PATH=...) that you must then copy and run separately. The document implies it's a single step. Someone following it will run pm2 startup, see text printed on screen, and not know they need to run that output as a second command. This needs a note.
The deployment pipeline on Oracle (git pull → pnpm install → pnpm build → pm2 restart) runs inside Oracle's SSH session via GitHub Actions. The deploy.yml needs to know the exact directory path on Oracle where the app was cloned — typically something like /home/opc/BharatScan. This path needs to be stored as a GitHub Secret (ORACLE_APP_DIR) alongside the SSH key and host, otherwise the deploy script doesn't know where to cd before running the commands.
Everything else is complete. This document is production-ready as a planning reference.
