Bug: syncSymbolMaster() in server/src/services/symbolMasterService.ts fails 
with HTTP 403 fetching https://public.fyers.in/sym_details/NSE_FO.csv (and 
likely NSE_CM.csv too). Root cause: the fetchText() calls for the two Fyers 
URLs pass no headers at all, while the NSE index fetches in the same file 
already send a full browser-like User-Agent + Referer specifically because 
"NSE archives requires a browser-like User-Agent + Referer to avoid 503s" 
(see the existing NSE_HEADERS constant and comment). Fyers' static file 
host appears to now enforce similar bot-protection against bare/non-browser 
requests — likely blocking Replit's and other cloud/datacenter IPs — and 
the code was never updated to send headers for those two calls.

Fix, in server/src/services/symbolMasterService.ts:

1. Reuse the existing NSE_HEADERS constant (or extract it to a shared 
   BROWSER_HEADERS name if that reads more clearly, given it's no longer 
   NSE-specific) for the two Fyers fetchText() calls:
     const foText = await fetchText(FYERS_FO_URL);
     const cmText = await fetchText(FYERS_CM_URL);
   → pass the same headers object used for NSE_INDICES fetches.

2. While you're in there: fetchText() currently has no timeout, so a 
   hanging connection (rather than a clean HTTP error) would block server 
   startup indefinitely inside bootstrapSymbolMasterIfEmpty(). Add a 
   reasonable timeout (e.g. 15 seconds) via AbortSignal.timeout(15000) 
   passed into the fetch() call inside fetchText(), so a network hang 
   fails fast with a clear error instead of hanging the bootstrap step.

3. Surface bootstrap failures in the UI, not just server logs: the 
   Settings → API / Data Source → "Symbol Master" card currently only 
   shows "Runs automatically every Monday at 7 AM IST" with no indication 
   of failure. Add a way for the frontend to know the last sync attempt's 
   outcome — e.g. persist the last bootstrap/refresh result (success + 
   timestamp + upserted count, or failure + timestamp + error message) 
   somewhere queryable (a simple table, or even a small JSON file/row is 
   fine — match whatever pattern sync_log or similar already uses 
   elsewhere in this codebase), and show it in that Settings card so a 
   failed sync is visible without digging through server logs. Reuse 
   whatever "last run status" pattern already exists for the nightly sync 
   jobs dashboard if there is one, for consistency.

Do not change the retry behavior (bootstrap on empty table + weekly 
Monday cron + manual refresh button all stay as-is) — this is purely 
about (a) making the actual network call succeed, and (b) making a 
future failure visible instead of silent.
