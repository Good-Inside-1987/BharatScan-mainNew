In server/src/adapters/fyers.ts, getOptionChain() builds the options-chain-v3
request URL but never sets the `strikecount` parameter, which Fyers'
official API requires on every request to this endpoint (confirmed via
multiple real usage examples referencing Fyers' own docs, e.g.:
  { "symbol": "NSE:NIFTY50-INDEX", "strikecount": 50, "timestamp": "" }
  { "symbol": "NSE:TCS-EQ", "strikecount": 1, "timestamp": "" }
This is almost certainly the actual cause of the "Please provide valid
inputs" error occurring for every underlying, including NIFTY and BANKNIFTY,
100% of the time regardless of network/rate-limit conditions.

Current code (around line 246-248):
    const url = new URL(`${FYERS_DATA_BASE}/options-chain-v3`);
    url.searchParams.set("symbol", underlying);
    if (expiry) url.searchParams.set("timestamp", expiry);

Fix:
1. Add a `strikecount` query param. Reuse whatever strike-range/count config
   already exists elsewhere in the codebase for options (search for
   `strikeRange` in optionsDataService.ts — reuse that same range value
   converted to a strike count, or default to 50 if no existing config
   value is a clean fit — do not invent a new unrelated constant).
2. Change the `timestamp` param to always be sent as an empty string ""
   rather than the ISO date string (e.g. "2026-07-21") currently passed in
   as `expiry` — every confirmed-working example in Fyers' own community
   sends empty string for this field, and there's no confirmed correct
   format for targeting a specific future expiry via this param. Keep the
   `expiry` argument in the function signature (other code calls it with
   a specific date), but log a one-time warning comment noting that
   targeting a specific expiry via `timestamp` is unconfirmed and currently
   ignored — do not silently drop this limitation without a comment.
3. Do not change response parsing, expiry extraction logic, or any other
   function in this file.
