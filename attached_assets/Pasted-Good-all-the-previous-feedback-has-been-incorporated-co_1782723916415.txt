Good — all the previous feedback has been incorporated correctly. This version is solid. A few remaining gaps:
Still Missing from Phase 3 — Will Block Auto-Deploy
GitHub Actions SSH key setup is completely absent. For GitHub Actions to connect to Oracle and run the deploy commands, you need to generate an SSH key pair, add the public key to Oracle's ~/.ssh/authorized_keys, and store the private key in GitHub Secrets. Without this, the deploy.yml workflow file exists but has nothing to connect with. This is a required step between "Create GitHub Actions workflow" and "Run first-run backfill."
Oracle has two firewall layers, not one. The document mentions Oracle's Security List (dashboard level) ✅, but Ubuntu itself also has its own firewall (UFW/iptables). Both block incoming traffic independently. Opening the Oracle Security List but not the OS firewall still results in no browser access. Need to run sudo ufw allow <port> on the Oracle terminal too.
Phase 1 Item That Isn't Actually Phase 1
attached_assets path is listed as "verify during Phase 3 setup." But the @assets Vite alias failing doesn't produce missing images — it fails the entire pnpm build and nothing deploys. This needs to be confirmed and fixed in Phase 1 before Oracle is ever attempted, not discovered during Oracle setup.
Backup Detail Missing
SQLite WAL mode (already enabled in the code) creates three files alongside each other: bharatscan.db, bharatscan.db-shm, and bharatscan.db-wal. The nightly backup must copy all three together as an atomic unit. Backing up only the .db file and restoring it alone can produce a corrupt database. The backup script needs to explicitly handle this.
Clarification Needed — post-merge.sh vs GitHub Actions
The document rewrites post-merge.sh in Phase 1 and creates deploy.yml GitHub Actions in Phase 3. These do the same job. On Replit, post-merge.sh was the hook for git pulls. On Oracle, GitHub Actions replaces this entirely. The document doesn't clarify this — it should state that post-merge.sh is only needed for Replit's workflow, and Oracle uses Actions instead. Otherwise someone will try to maintain both.
One Factual Concern
The roadmap says MarketApiPanel connects to an external API URL. After Phase 2 builds the server-side data layer, those endpoints (/stocks/all, /options/all, /stocks/since/:date) will live on the Oracle server itself. The panel's URL input needs to either be pre-configured to point to localhost server-side, or the entire panel becomes redundant and gets replaced by automatic sync. This transition isn't addressed anywhere — what happens to that UI component after the Angel API layer is built?
