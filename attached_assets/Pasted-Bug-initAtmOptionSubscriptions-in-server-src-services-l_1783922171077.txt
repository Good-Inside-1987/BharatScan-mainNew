Bug: initAtmOptionSubscriptions() in server/src/services/liveOptionsTracker.ts 
calls adapter.getOptionExpiries() for every configured index on every 
invocation, with no per-day caching — since reconcileLiveFeedState() now 
calls this on every restart during market hours (not just once at a fixed 
9 AM), a day with several restarts (Replit sleep/wake, redeploys, manual 
testing) triggers redundant Fyers API calls that add nothing, since the 
correct expiry for a given underlying does not change within a trading day.

Fix, in server/src/services/liveOptionsTracker.ts:

1. Add a small persisted cache — reuse the same one-row-per-day pattern 
   already used for request_budget in marketDataService.ts and holiday/ 
   symbol bootstraps — e.g. a table `atm_expiry_cache(underlying TEXT, 
   date TEXT, expiry TEXT, PRIMARY KEY(underlying, date))` in market.db.

2. In initAtmOptionSubscriptions()'s per-underlying loop, before calling 
   getExpiries(), check the cache for today's date + this underlying. If 
   present, use the cached expiry directly and skip the API call. If 
   absent, call getExpiries() as today, pick the nearest expiry as today, 
   and persist it to the cache before continuing.

3. Do NOT cache getOptionChain() the same way — that call is what 
   determines the current ATM strike from the live spot price, which does 
   change through the day, so it should still be called fresh on every 
   restart to get an accurate ATM window right now. Only the expiry lookup 
   is the redundant part.

4. Leave subscribeSymbols()/WS resubscription behavior completely 
   unchanged — that genuinely needs to run on every restart since Fyers' 
   server has no memory of prior subscriptions once a connection drops.

After this, verify: restart the server twice in a row during market hours 
and confirm the second restart's logs show the cached expiry being reused 
("[liveOptionsTracker] Using cached expiry {expiry} for {underlying} 
(already resolved today)" or similar) rather than a fresh getOptionExpiries 
call, while option/stock subscriptions still rebuild correctly both times.
