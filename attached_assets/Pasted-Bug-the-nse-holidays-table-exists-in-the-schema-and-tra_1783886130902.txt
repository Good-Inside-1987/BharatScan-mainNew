Bug: the nse_holidays table exists in the schema and tradingCalendar.ts's
isHoliday()/isTradingDay() are already built to consult it, but nothing
anywhere in the codebase ever writes a single row into it — no scraper,
no seed script, no manual import UI. isHoliday() degrades gracefully to
"not a holiday" when the table is empty, which was a reasonable
fallback, but it means every NSE trading holiday that falls on a weekday
(Republic Day, Holi, Diwali, etc.) is silently treated as an ordinary
trading day by the nightly cron jobs and by the catch-up scheduler —
wasting a full day's worth of request budget fetching data for a date
the market was actually closed, on exactly the days that budget is
scarcest to waste, and risking bogus rows in ohlcv_daily/etc. if Fyers'
API returns stale/previous-day data for a non-trading date instead of an
empty response.

There's a second, related bug this fix must also correct:
tradingCalendar.ts's isHoliday() gates on a cached
"is the table populated at all" check:

    let holidaysPopulated: boolean | null = null;
    function nseHolidaysPopulated(): boolean {
      if (holidaysPopulated !== null) return holidaysPopulated;
      const row = marketDb.prepare(`SELECT 1 FROM nse_holidays LIMIT 1`).get();
      holidaysPopulated = !!row;
      return holidaysPopulated;
    }

Once this caches `false` (which it will, the first time isHoliday() is
called against today's still-empty table — e.g. during startup
catch-up), it never re-checks for the rest of that process's lifetime.
So even after this fix successfully populates nse_holidays moments
later, isHoliday() would keep returning false forever for that running
process, silently undoing the fix. This check is also just redundant:
`SELECT 1 FROM nse_holidays WHERE date = ?` already correctly returns no
rows (i.e. "not a holiday") when the table is empty, with no separate
existence check needed.

Fix, in two parts:

PART 1 — server/src/services/tradingCalendar.ts: remove the cache

Delete the `holidaysPopulated` variable and `nseHolidaysPopulated()`
function entirely, and simplify isHoliday() to query directly:

    export function isHoliday(dateStr: string): boolean {
      const row = marketDb.prepare(`SELECT 1 FROM nse_holidays WHERE date = ?`).get(dateStr);
      return !!row;
    }

This table will only ever hold a few dozen rows, so there's no
performance concern in dropping the cache — and this removes the
stale-cache bug as a side effect.

PART 2 — new file server/src/services/holidayCalendarService.ts: fetch
and persist the real calendar

NSE publishes the official trading holiday list at a public JSON
endpoint: https://www.nseindia.com/api/holiday-master?type=trading —
the same nseindia.com API family already used by
supplementaryJobs.ts's FII/DII and PE/PB/DivYield jobs (which need the
same session-cookie handshake to avoid being blocked). Check whether
extracting supplementaryJobs.ts's getNseSessionCookies() and
nseApiGet<T>() into a small shared module (e.g. nseSession.ts) is a
clean, low-risk change — if so, prefer that and have both
supplementaryJobs.ts and this new file import from it; if it'd require
touching more than those two call sites, it's fine to duplicate the two
small helper functions into the new file instead, matching this
project's existing "correctness over premature abstraction" bias (see
Prompt 15's node_modules note for the same reasoning).

The endpoint's exact response shape should be verified against the live
call rather than assumed, since NSE's API responses have drifted before
and this project doesn't have a way to test against the real endpoint
before you apply it — log the raw JSON on the first real run and adjust
field names if they don't match. Best current understanding of the
shape: a top-level object keyed by segment ("CM" for the equity/cash
market, "FO" for derivatives, plus others), each segment an array of
objects with fields resembling `tradingDate` (e.g. "26-Jan-2026") and
`description`. Union the CM and FO segments' dates (equity and F&O
holidays are effectively identical in practice, but this avoids relying
on that assumption), parse each date into ISO (YYYY-MM-DD) format, and
de-duplicate.

Write a function:

    export async function syncNseHolidayCalendar(): Promise<{ upserted: number }> {
      // 1. getNseSessionCookies() → nseApiGet("/api/holiday-master?type=trading", cookies)
      // 2. Union + parse CM and FO segment dates to ISO strings
      // 3. Upsert each into nse_holidays:
      //      INSERT INTO nse_holidays (date, description) VALUES (?, ?)
      //      ON CONFLICT(date) DO UPDATE SET description = excluded.description
      // 4. Return { upserted: <count> }
    }

Wire it in exactly the same places, and the same way, symbolMaster
already is (it's the closest existing analogue — public data, no broker
auth needed, safe to run unconditionally, needs a startup bootstrap so a
fresh deployment isn't stuck for up to a week):

1. In server/src/config/environment.ts's syncSchedule, add:
     holidayCalendar: "15 7 * * 1",  // 7:15 AM IST every Monday — right after symbolMaster
2. In server/src/services/scheduler.ts, add a bootstrapHolidayCalendarIfEmpty()
   function mirroring bootstrapSymbolMasterIfEmpty() (empty-table check on
   nse_holidays, calls syncNseHolidayCalendar(), logs success/failure,
   never throws), call it from registerAllCronJobs() right alongside the
   existing bootstrapSymbolMasterIfEmpty() call — before
   runStartupCatchUp(), since catch-up's isTradingDay() checks depend on
   this data being current as early as possible.
3. Add a cron.schedule(config.syncSchedule.holidayCalendar, …) block in
   registerAllCronJobs(), same pattern as the existing symbolMaster cron
   block (log start, call syncNseHolidayCalendar(), log result/error).
4. Add it to getSchedulerStatus()'s `jobs` object the same way
   symbolMaster is listed, so the Settings page's scheduler status
   reflects it too.

Do not change anything about how FII/DII or PE/PB/DivYield are fetched —
this is purely additive, following their existing pattern for a new,
independent data source.

Test: with an empty nse_holidays table, restart the server and confirm
the bootstrap logs a successful sync with a non-zero row count; spot-check
a couple of known 2026 NSE holidays that fall on a weekday (e.g. Republic
Day, Jan 26) actually appear in the table afterward; then confirm
isTradingDay() returns false for one of those dates and true for an
ordinary weekday immediately after — in the same process, with no
restart — to verify the Part 1 cache fix actually resolves the
stale-cache issue and not just the empty-table one.
