# Prompt 25 — Stop retrying a genuinely dead token; fix misclassified auth errors

## What the fresh log proved
Every symbol in the run failed with `"Could not authenticate the user"` —
100% failure this time, not the mixed pattern from before. The live feed
bridge's raw (unmodified) error in the same log confirms this is a REAL
expired token, not a rate-limit quirk:

```
[liveFeedService] Bridge control: {"__control__": "error", "detail":
  "{'type': 'cn', 'code': -99, 'message': 'Token is expired', 's': 'error'}"}
```

Prompt 24's retry logic treated `"Could not authenticate the user"` as
transient overload (based on an earlier, incorrect theory) and retried it
twice per symbol with backoff — wasted time/requests against a session that
was never going to succeed, across potentially thousands of symbols.

## Root cause
`classifyAdapterError()` in `server/src/services/marketDataService.ts` is
supposed to turn this exact message into a `SessionExpiredError` (which the
sync loop already knows to treat as fatal-and-stop, not retry) — but its
regex didn't include Fyers' actual wording, so the message fell through to a
generic `BrokerUnavailableError`, which `syncJobs.ts`'s retry logic then
matched against its (now known to be wrong) transient-pattern list instead.

## Fix — two files

### File 1: `server/src/services/marketDataService.ts`
Find:
```ts
  if (/not configured|session|expired|unauthori[sz]|invalid.?(token|key|credentials)|forbidden/i.test(message))
    throw new SessionExpiredError(message);
```
Replace with:
```ts
  if (/not configured|session|expired|unauthori[sz]|invalid.?(token|key|credentials)|forbidden|could not authenticate/i.test(message))
    throw new SessionExpiredError(message);
```

### File 2: `server/src/services/syncJobs.ts`
Find the `TRANSIENT_OVERLOAD_PATTERNS` block (added in Prompt 24) and its
comment, and replace the whole block with:
```ts
// Fyers returns an HTML page instead of JSON when it's rate-limiting a
// request on an otherwise-valid session — already normalized by fyers.ts's
// fetchJson() into Error("Rate limited by Fyers (received HTML instead of
// JSON)"). This is genuinely transient and worth a short backoff + retry.
//
// NOTE: an earlier version of this list also treated Fyers' literal
// "Could not authenticate the user" message as transient, on the theory
// that Fyers uses that auth-sounding wording as a generic overload signal.
// Real-world logs disproved that — it corresponded to an actually-expired
// token (confirmed independently via the live feed bridge's raw error:
// {'code': -99, 'message': 'Token is expired'}). That message is now
// classified upstream in marketDataService.ts's classifyAdapterError() as
// a real SessionExpiredError, which this loop already handles correctly by
// stopping the whole job cleanly instead of burning retries/budget hammering
// a dead token across thousands of symbols.
const TRANSIENT_OVERLOAD_PATTERNS = [
  /rate limited by fyers/i,
];
```
(No other code in `syncJobs.ts` needs to change — `fetchBarsWithRetry()`
already has `if (err instanceof AuthenticationError || err instanceof
SessionExpiredError) throw err;` before the transient-pattern check, so once
`classifyAdapterError` correctly reclassifies the message, this path already
does the right thing.)

## Before re-testing this
Your Fyers token is dead right now — **re-authenticate/re-login to Fyers
first**, independent of this code change. No code fix repairs an already-
expired token.

## Verify after applying + re-authenticating
1. `grep -n "could not authenticate" server/src/services/marketDataService.ts`
   should show it inside the `SessionExpiredError` regex.
2. `grep -n "could not authenticate" server/src/services/syncJobs.ts` should
   return **nothing** (it's been removed from the transient-pattern list).
3. Run a sync with a valid, fresh token — should see normal completions, no
   `"Transient overload"` lines for auth messages.
4. To confirm the fix actually triggers correctly for a truly-expired token
   (optional, harder to test deliberately): if the token expires mid-run
   again, you should see `"[syncJobs] Broker session unavailable mid-job"`
   and the loop stopping immediately — not hundreds of individual
   `"Transient overload... retrying"` lines burning through the whole
   symbol list.
