Root cause confirmed: in server/src/services/marketDataService.ts, upsertBars() writes 
whatever date string the Fyers adapter returns for daily bars (a full ISO timestamp, e.g. 
"2026-07-23T00:00:00.000Z") directly into ohlcv_daily.date. queryBars() then filters with
`date BETWEEN ? AND ?` using plain "YYYY-MM-DD" bounds. Since SQLite does string comparison
on TEXT columns, a full ISO timestamp is lexicographically greater than the plain date string
with the same prefix, so `date BETWEEN '2026-07-23' AND '2026-07-23'` never matches a row
stored as '2026-07-23T00:00:00.000Z'. This silently makes every same-day query (exactly what
the EOD sync loop always does, since it always passes from=to=date) return an empty array
even though the row was just written successfully — which is why stats.completed stayed at
0 while stats.failed climbed to the full symbol count, despite hundreds of visible successful
"✓ inline fetch ... (1 bars)" log lines.

Fix (two parts):

1. Going forward — normalize at write time, not read time:
   In upsertBars() in marketDataService.ts, for the ohlcv_daily branch (isDaily(resolution)),
   normalize b.date to a plain YYYY-MM-DD string before inserting:
     const plainDate = b.date.slice(0, 10);
     stmt.run(symbol, plainDate, b.open, b.high, b.low, b.close, b.volume);
   Do NOT change the ohlcv_intraday branch — intraday timestamps legitimately need full
   date+time precision, only the daily table has this mismatch.

2. Migrate existing bad rows — the DB already has real data stored as full ISO timestamps
   (confirmed: SELECT DISTINCT date FROM ohlcv_daily currently returns values like
   '2026-07-22T00:00:00.000Z' instead of '2026-07-22'). Add a one-time idempotent migration
   (run at startup, guarded so it only runs once) that normalizes any existing ohlcv_daily.date
   values containing 'T' down to their first 10 characters:
     UPDATE ohlcv_daily SET date = substr(date, 1, 10) WHERE date LIKE '%T%';
   Log how many rows were normalized. Run this BEFORE any sync jobs execute on startup so
   isEodCovered() and queryBars() see corrected data immediately, not after the next sync.

3. After applying, please remove the temporary debug logging added in Prompt 26 
   (the completedBefore/assertion block and the periodic-tally console.log calls in 
   runSymbolLoop) — that diagnostic served its purpose and confirmed this exact root cause,
   no need to keep it running in production.

Please verify: after this fix, re-run the app and confirm sync_log shows symbols_completed
close to the total symbol count (minus genuinely delisted/no-data symbols) instead of 0.
