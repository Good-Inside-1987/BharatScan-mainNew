BharatScan — Complete Architecture, Analysis & Roadmap (Definitive Final Version)
WHAT BHARATSCAN IS
A Chartink-inspired stock screener and backtesting tool for Indian markets (NSE/BSE). Built with React 19 + TypeScript frontend, Express + Node.js backend, SQLite database. Currently runs on Replit. Goal: fully automatic data (no CSV uploads ever), deployed on Oracle Cloud, eventually open to paying users.

CURRENT PROBLEMS BEING SOLVED
User must upload CSV files every time the app opens — heavy and slow
All market data lives only in browser memory — refresh = data gone
App only works when Replit editor is open — no always-on hosting
No automatic data fetching — everything is manual
CHOSEN SOLUTION STACK
Data Source: Angel One SmartAPI (free with Angel One broker account)
Hosting: Oracle Cloud Free Tier — ARM Ampere A1 (4 CPU, 12–24GB RAM — verify actual number at Oracle sign-up, may be 12GB not 24GB; 12GB is sufficient for single-user SQLite + Node/Express; 200GB storage — free forever)
Secure URL: Cloudflare Tunnel (free HTTPS, hides server IP, DDoS protection)
Code Management: GitHub → auto-deploy to Oracle via GitHub Actions (main branch only)
Process Manager: PM2 — three named processes: bharatscan (Express web server), bharatscan-scheduler (Central Scheduler), bharatscan-tunnel (Cloudflare Tunnel)
Database: Two separate SQLite files (market.db + app.db) → PostgreSQL when going public
Alerts: Telegram Bot API (HTTPS POST — simple, reliable; cloud providers commonly block outbound SMTP on free tier so email is unreliable)
Total cost: ₹0

ANGEL ONE API SETUP
Create 2 API keys from same Angel One account (2 separate app registrations):
Key 1 → Oracle production server
Key 2 → Replit development environment (prevents session conflicts)
TOTP auto-refresh is critical: Angel session token expires daily. Must extract raw TOTP secret from Angel's QR code during 2FA setup — before completing setup (cannot retrieve after). This secret auto-generates daily login codes. Without it, 4 PM sync silently fails every day.
Rate limiting: Fetching 2000+ stocks in a loop hits Angel's API call limits. All sync code must throttle calls and include retry logic.
Intraday history limit: Angel API provides approximately 30 days of 1-min historical data (not 60–90 days). Verify exact limit from Angel API documentation before writing the backfill script. Intraday accumulates from setup day forward permanently.
DATABASE DESIGN — TWO SEPARATE FILES
market.db — all market data (large, re-fetchable if lost)

Stock OHLCV (daily + intraday), options data (indices + stocks)
FII/DII data, PE ratios, MF holdings
Symbol master, sectors, lot sizes, holiday calendar
app.db — all user data (small, critical, cannot be recreated)

Saved scans, portfolio, alerts, paper trades, simulation history
User settings, Angel API credentials, app metadata
Why separate: If a large market data sync corrupts market.db, user data in app.db is completely safe. Market data is re-fetchable. User data is irreplaceable.

Migration note: Existing bharatscan.db must be migrated. A migration script runs at the start of Phase 2A to move all existing user data into new app.db before the split. Without this, all Phase 1 data is silently abandoned.

Code change scope: server/src/db.ts currently exports a single database connection used by every route file. After the split, every route file (scans.ts, portfolio.ts, alerts.ts, settings, etc.) must be updated to import from the correct database — marketDb or appDb. This is a full server refactor, not just creating two files.

PM2 PROCESS ARCHITECTURE — THREE NAMED PROCESSES
ecosystem.config.js defines:
bharatscan           → Express web server (app + API routes)
bharatscan-scheduler → Central Scheduler (all sync jobs — completely separate process)
bharatscan-tunnel    → Cloudflare Tunnel
Deploy restarts:  pm2 restart bharatscan ONLY
                  Scheduler keeps running uninterrupted during deploys
                  Tunnel keeps running uninterrupted during deploys
Scheduler update: pm2 restart bharatscan-scheduler (deliberate, separate step)

Why the scheduler must be separate: If the scheduler lives inside the Express process, every routine pm2 restart bharatscan from a deploy kills the scheduler mid-run. A 4 PM EOD sync across 2000+ stocks interrupted at 4:20 PM by a deploy leaves that day's data incomplete with no error. With a separate process, deploys never touch the scheduler.

SQLITE BACKUP — CRITICAL DETAILS
WAL mode creates 3 files per database — all 6 must be handled correctly:

market.db + market.db-shm + market.db-wal
app.db + app.db-shm + app.db-wal
Plain file copy (cp) is not safe while the database is running. Correct method:

Use SQLite's built-in .backup command, OR
Run PRAGMA wal_checkpoint(FULL) first to flush WAL into main .db file, then copy
Never use plain cp on live database files.

TIMEZONE — CRITICAL FOR ALL SYNC JOBS
Oracle Ubuntu defaults to UTC. India is UTC+5:30 (IST).

If the scheduler's cron expressions are not explicitly set to Asia/Kolkata, "4:00 PM" fires at 4:00 PM UTC = 9:30 PM IST — hours after market close.

Fix: set Oracle server timezone (timedatectl set-timezone Asia/Kolkata) during Phase 0 setup AND make all scheduler cron expressions explicitly timezone-aware using Asia/Kolkata. Both required — one as safety net for the other.
