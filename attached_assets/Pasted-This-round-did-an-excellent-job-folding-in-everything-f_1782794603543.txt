One significant thing is missing though, and it's specifically what I recommended two messages ago, so it's worth flagging clearly rather than letting it slide.
The phase sequencing doesn't reflect what we just discussed. Phase 3 is still written as a single block that comes after Phase 1 and all of Phase 2A through 2D are finished. Nowhere does it say to start the Oracle account and instance creation now, independent of code readiness. That matters concretely: the 7-day idle-reclaim policy means if you grab the instance and then don't touch it again until you're done with weeks of Phase 1/2 work, Oracle can stop it for inactivity before you ever reach Phase 3 — and the "out of host capacity" problem means re-acquiring it later isn't guaranteed to be quick. As written, this document would have you start the account/instance hunt only after finishing all the application code, which is exactly the sequencing I argued against.
The fix doesn't require renumbering anything — just pull a handful of Phase 3 items out and run them as a parallel track starting today: create the Oracle account, start retrying instance creation (worth noting in the doc itself that this step alone can take retries spanning hours to days depending on region capacity, so it shouldn't block on anything else), then once it exists, install Node, pnpm, and PM2, and get a trivial placeholder process running under PM2 — it doesn't need to be the real app, just something with real CPU/network activity so the instance doesn't read as idle. Wire up Cloudflare Tunnel and the GitHub Actions SSH access against that placeholder. None of this depends on Phase 1 or Phase 2 being finished. The genuinely Angel-API-dependent parts of Phase 3 — the Day 1 backfill, holiday calendar, starting the Central Scheduler — correctly stay where they are, at the end, since those do need the application code to actually exist.
The Central Scheduler needs its own PM2 process, separate from the web app. This connects directly to the point above. If the scheduler that runs the 4 PM EOD sync, intraday sync, options sync, and nightly cleanup lives inside the same Node process as the Express server — which is the natural way to build it — then every pm2 restart bharatscan triggered by a deploy also kills the scheduler mid-run. Given this whole pipeline is built around fast, frequent deploys (1-2 minutes per push), and given the parallel-track approach above means deploys will start happening well before Phase 2 is finished, this collision becomes a real risk rather than a theoretical one: push a small fix at 4:15 PM IST while the EOD sync for 2000+ stocks is mid-flight, and the restart silently kills it partway through with no retry. The fix is straightforward — run the scheduler as its own named PM2 process (something like bharatscan-scheduler), entirely separate from bharatscan. App deploys restart the web server only; the scheduler keeps running undisturbed unless its own code changes.
Auto-rollback needs something concrete to roll back to. The deploy pipeline diagram says a failed health check triggers git reset to the previous build, but nothing in the script as described actually captures what that previous commit was before pulling new code. The deploy script needs to store git rev-parse HEAD into a variable right before git pull, so the rollback branch has an actual commit hash to reset to rather than an undefined target.
Outside of these three, the document is genuinely ready to execute. The sequencing point is the one I'd act on first, since it's the only one that doesn't wait on any more planning — it's something you can start today, separately from whatever Replit AI does next on the code.
