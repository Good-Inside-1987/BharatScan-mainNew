Two things, one of them a real catch on the idle-reclaim fix itself, the other a sharp internal inconsistency in the document.
The Phase 0 idle-reclaim fix probably doesn't actually work. A Node script that prints "alive" once a minute spends nearly all of its time doing nothing between prints — the print itself is sub-millisecond. Oracle's own reclaim criteria looks at 95th-percentile CPU utilization staying under roughly 20% over a 7-day window, plus similarly low thresholds for network and memory on A1 shapes specifically. A once-a-minute console.log will sit at effectively 0% CPU and 0% network for the entire window — it's almost the textbook definition of what that policy is built to catch, not what would clear it. As written, Phase 0 would have you go through the hassle of acquiring the instance, then lose it to idle reclaim anyway despite believing you'd covered it.
The cleaner fix removes the placeholder concept entirely rather than trying to make it more convincing: pull forward just three specific Phase 1 items — the Replit vite plugin guard (#1), the production Express static serving and SPA catch-all route (#3 and #4) — so there's a minimal but real, production-buildable version of the actual app, even with none of the Angel API or database-split work done yet. Deploy that to the Oracle box in Phase 0 under a single bharatscan PM2 process, point Cloudflare Tunnel at it (which Phase 0 currently defers to Phase 3 for no real reason — tunnel setup doesn't depend on app completeness either), and have something external hit it periodically — a free uptime monitor like UptimeRobot pinging /api/health every five minutes works well and costs nothing. That generates genuine, recurring, externally-driven network traffic, which is a far more reliable idle-reclaim signal than a local print loop, and it means the deploy pipeline gets real validation from day one instead of being tested for the first time at the end of Phase 3. One thing worth avoiding regardless: don't reach for deliberately burning CPU with something like stress-ng to game the threshold — that approach shows up repeatedly in community discussions as a real risk of account suspension under Oracle's Always Free terms, not a safe workaround. This restructure also quietly resolves the separate question of cleaning up the placeholder process later — there isn't one to clean up, since the real bharatscan process just keeps running and gets promoted in place when the scheduler and tunnel processes join it in Phase 3.
Phase 3's own deploy.yml checklist is missing a step that's in the pipeline diagram right above it. The "DEPLOYMENT PIPELINE" section earlier in the document correctly lists pnpm build between pnpm install --frozen-lockfile and pm2 restart bharatscan. The Phase 3 checklist's restatement of the same deploy.yml drops that line entirely — it goes straight from pnpm install --frozen-lockfile to pm2 restart bharatscan by name only. If the actual workflow file gets built off the Phase 3 checklist rather than the diagram, every deploy would restart PM2 against whatever was already sitting in dist/ from the last successful build, never compiling the newly pulled code. The health check would still pass — the old build is presumably still healthy — so it would report "Deploy successful" via Telegram while silently shipping nothing. Needs pnpm build added back explicitly in that checklist item, between install and restart.
Everything else holds up — the database design, the timezone handling, the rollback logic's PREV_COMMIT capture, the three-process PM2 split. Once these two are folded in, this is ready to act on. The first one is worth deciding before you start Phase 0 today, since it changes what you're actually deploying in that first step rather than something you'd patch later.
