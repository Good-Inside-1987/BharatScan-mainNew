# Prompt 24 — Retry Fyers' transient overload errors instead of failing immediately

## What the logs showed
After Prompt 23, a fresh catch-up run against 2964 symbols for 2026-07-17
(broker authenticated, confirmed working) failed on hundreds of symbols —
but with two *different* error messages randomly interleaved across
adjacent, alphabetically-ordered symbols in the same run:

- `Could not authenticate the user`
- `Rate limited by Fyers (received HTML instead of JSON)`

Both are literal responses from Fyers itself (see `fyers.ts` — `data.message`
for the JSON case, and the HTML-detection branch in `fetchJson()` for the
other). Fyers is known to return `"Could not authenticate the user"` as a
generic overload signal even on a valid, already-authenticated session when
requests arrive too fast — it is not a real auth failure (those come through
the app as `AuthenticationError`/`SessionExpiredError` instead, which this
code already distinguishes and handles separately).

Random alternation between these two messages on a symbol-by-symbol basis,
for names right next to each other alphabetically, is the signature of
per-second rate throttling, not a data/symbol problem — the existing 150ms
pacing isn't leaving enough headroom.

## Fix — two parts, both in `server/src/services/syncJobs.ts`

### Part 1: raise the base delay
Find:
```ts
      await new Promise((resolve) => setTimeout(resolve, 150));
      const bars = await getHistoricalBars(toFyersSymbol(symbol), resolution, date, date);
```
Replace with:
```ts
      await new Promise((resolve) => setTimeout(resolve, 300));
      const bars = await fetchBarsWithRetry(symbol, resolution, date);
```

### Part 2: add a retry-with-backoff helper
Add this new block directly above `async function runSymbolLoop(` (keep the
existing JSDoc comment on `runSymbolLoop` where it is):

```ts
// Fyers surfaces overload/rate-limiting two different ways under load:
//   1. An HTML page instead of JSON — already normalized by fyers.ts's
//      fetchJson() into Error("Rate limited by Fyers (received HTML instead of JSON)").
//   2. A well-formed JSON error with message "Could not authenticate the user" —
//      this is Fyers' own documented quirk: their gateway returns this exact
//      auth-sounding message when a valid, already-authenticated session is
//      simply being throttled for sending requests too fast. It is NOT a real
//      session/token problem (those come through as AuthenticationError /
//      SessionExpiredError from marketDataService.ts instead).
// Both are almost always transient — a short backoff and retry succeeds.
const TRANSIENT_OVERLOAD_PATTERNS = [
  /rate limited by fyers/i,
  /could not authenticate the user/i,
];

function isTransientOverloadError(err: unknown): boolean {
  const message = err instanceof Error ? err.message : String(err);
  return TRANSIENT_OVERLOAD_PATTERNS.some((re) => re.test(message));
}

async function fetchBarsWithRetry(
  symbol: string,
  resolution: string,
  date: string,
  maxRetries: number = 2
): Promise<Awaited<ReturnType<typeof getHistoricalBars>>> {
  let lastErr: unknown;
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      return await getHistoricalBars(toFyersSymbol(symbol), resolution, date, date);
    } catch (err) {
      lastErr = err;
      if (err instanceof AuthenticationError || err instanceof SessionExpiredError) throw err;
      if (!isTransientOverloadError(err) || attempt === maxRetries) throw err;
      const backoffMs = 1000 * (attempt + 1); // 1s, then 2s
      console.warn(
        "[syncJobs] Transient overload syncing %s @ %s (attempt %d/%d) — retrying in %dms: %s",
        symbol, date, attempt + 1, maxRetries + 1, backoffMs,
        err instanceof Error ? err.message : String(err)
      );
      await new Promise((resolve) => setTimeout(resolve, backoffMs));
    }
  }
  throw lastErr;
}
```

`AuthenticationError` and `SessionExpiredError` are already imported at the
top of the file — no new imports needed.

## Verify after applying
1. `grep -n "fetchBarsWithRetry\|TRANSIENT_OVERLOAD_PATTERNS" server/src/services/syncJobs.ts`
   should show the new helper and its usage inside the loop.
2. Restart the app and let a catch-up run happen (or trigger one manually).
   In the log, a healthy sign is seeing occasional
   `[syncJobs] Transient overload syncing X @ Y (attempt 1/3) — retrying in 1000ms: ...`
   lines followed by that same symbol NOT appearing in a final `"Failed to
   sync"` line — meaning the retry succeeded.
3. Compare the final `[syncJobs] EOD sync done — status=... completed=... failed=...`
   counts against the previous run — `failed` should drop substantially.
4. If `failed` is still high after this, the next thing to check is whether
   another process (live feed reconciler, options sync, or the frontend's
   10-second quote polling) is firing Fyers requests concurrently with this
   job and eating into the same real-time rate limit — that would need a
   shared cross-job request gate, not just tighter pacing inside this one loop.
